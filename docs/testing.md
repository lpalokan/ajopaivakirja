# Testing guide

This project is **BDD-first**: every feature and bug fix starts as a
plain-English Gherkin scenario that runs on an Android emulator. This
document explains how the suite is laid out and how to maintain it.

## Layers

| Layer | Where | Runs on | Command |
|------|-------|---------|---------|
| Unit / widget | `test/` | Host Dart VM | `flutter test` |
| End-to-end (Gherkin) | `integration_test/features/` | Android emulator | `./scripts/integration-report.sh` |
| Boot smoke | `integration_test/app_smoke_test.dart` | Android emulator | `flutter test integration_test/app_smoke_test.dart` |

The Gherkin layer is the **test-case database**: the `.feature` files are
the authoritative, human-readable catalogue of what the app does.

## Directory layout

```
integration_test/
  features/
    *.feature              # Gherkin scenarios (the source of truth)
    *_test.dart            # GENERATED by build_runner (git-ignored)
    step/
      *.dart               # step definitions (one phrase per file)
  support/
    harness.dart           # the engine: fakes + reusable actions
  app_smoke_test.dart      # standalone device boot check
build.yaml                 # bdd_widget_test configuration
```

`bdd_widget_test` reads each `.feature`, generates a matching
`*_test.dart`, and looks up each step in `features/step/`. The generated
files are git-ignored and rebuilt on every run, so **never edit a
`*_test.dart`** — edit the `.feature` or the step/harness instead.

## The BDD-first loop (mandatory — see CLAUDE.md)

1. **Write the scenario** in the relevant `*.feature` (or a new one).
2. **Reuse steps.** Check the catalogue below. Only add a new step if no
   phrase fits.
3. **Generate + run, expect red:**
   ```bash
   dart run build_runner build --delete-conflicting-outputs
   ./scripts/integration-report.sh
   ```
4. **Implement** the feature until the scenario is green; refactor.

## Writing scenarios

- Use `Background:` for setup. Every feature already starts with:
  ```gherkin
  Background:
    Given a clean database
    And the app is running
  ```
  This wipes the on-device SQLite DB and launches the real app with
  external services (notifications/location/Sheets/OCR) faked, so no
  permission dialogs or network are hit. Debug builds then seed two
  routes: **Töihin** (Koti→Työ, 54 km) and **Kotiin** (Työ→Koti, 54 km).
- Strings are double-quoted; whole numbers are bare (`{string}` / `{int}`).

## Step catalogue

| Gherkin phrase | What it does |
|---|---|
| `Given a clean database` | Truncates all SQLite tables |
| `Given the app is running` | Pumps the real app, waits for seeding |
| `When I open settings` / `routes` / `history` | Navigates from Home |
| `When I go back` | Taps the AppBar back button |
| `When I tap {string}` | Scrolls to and taps text |
| `When I enter {string} in the {string} field` | Types into a Settings field by label |
| `When I enter {string} in the dialog {string} field` | Types into a dialog field by label |
| `When I save settings` | Scrolls to and taps "Tallenna" (keeps SnackBar) |
| `When I start the {string} route at {int} km` | Opens route, fills odometer, confirms |
| `When I arrive at {int} km` | Taps "Olen perillä", confirms arrival odometer |
| `When I add route {string} from {string} to {string} of {int} km` | Adds a route via the dialog |
| `When I open the add route dialog` | Opens the new-route dialog |
| `When I swipe {string} left` / `right` | Dismissible swipe (delete / edit) |
| `When I tap the {string} dialog button` | Taps a dialog button by label |
| `When I sync to sheets` | Taps the History sync button |
| `When I toggle debug logging` | Flips the debug-logging switch |
| `Then I see {string}` | Asserts text is visible (scrolls if needed) |
| `Then I do not see {string}` | Asserts text is absent |
| `Then I see text containing {string}` | Substring assertion |

## Adding a new step

1. Add the phrase to a scenario, e.g. `When I clear the odometer field`.
2. Run `dart run build_runner build`. If the step is unknown,
   `bdd_widget_test` generates a stub file in `features/step/`.
3. Implement the stub as a one-liner delegating to `support/harness.dart`
   (add a function there if new behaviour is needed). Keep step files
   thin; put real logic in the harness so it is reusable.

`bdd_widget_test` derives the step file/function name from the phrase
(words camel-cased, `{placeholders}` dropped, parameters passed
positionally in order). If a hand-written step file name doesn't match
what the generator expects, the generator emits its own stub — copy the
one-line body across and delete the mismatched file.

## Running

```bash
# Host unit/widget tests (no emulator)
flutter test

# Full Gherkin suite on the emulator + a report to hand back
./scripts/integration-report.sh
```

`integration-report.sh` boots the `test_pixel` AVD if no device is
attached, runs `build_runner`, runs the suite, and writes
`reports/integration-report-<timestamp>.txt`.

## Troubleshooting

- **`build_runner` fails / wrong step name:** read its output; rename the
  step file/function to match, or fill the generated stub.
- **Flaky finder / timing:** prefer the harness `waitFor`, `settle`, and
  `scrollIntoView` helpers rather than bare `pump`. Transient SnackBars
  must be asserted right after a `pumpFor` step (e.g. `I save settings`),
  never after a settle.
- **Generated `*_test.dart` shows in git:** it's git-ignored; do not
  commit or edit it.
- **Integration binding:** because features live under `integration_test/`,
  the generated tests get `IntegrationTestWidgetsFlutterBinding`
  automatically. If a generated `main()` lacks it, set the binding via
  `build.yaml` and regenerate.
